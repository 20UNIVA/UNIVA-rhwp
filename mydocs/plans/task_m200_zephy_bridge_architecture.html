UNIVA-rhwp SSR Sub-1 — WebSocket 양방향 통신과 코드 레벨

네 객체 — LLM(vLLM/qwen3.5)·노트북·웹서버(rhwp-server)·프론트(rhwp-studio) — 사이의 통신을 단계별로 보여줍니다. 브라우저 ↔ 서버는 *WebSocket 양방향*으로 통합되어, *서버 → 클라 push*(LLM 명령 결과)와 *클라 → 서버 미러링*(사용자가 직접 친 편집) 둘 다 같은 채널을 사용합니다. 각 박스를 클릭하면 그 객체의 *재사용 기존 코드*와 *Sub-1에서 신규/수정되는 코드*가 두 칸으로 정리됩니다.

LLM의 역할

기존 — 재사용
  • vLLM (사내 serve-dev.rest.univa-internal.com) — qwen3.5 모델
  • 도구 호출 패턴 — Bash("hwp-doc-patch <action> --payload '...'") 형태 그대로
  • SYSTEM_PROMPT·SKILL.md 변경 0
왜? LLM 입장의 외양을 바꾸지 않기로 brainstorming에서 결정. 모든 변화는 *노트북 내부 라우팅*에서 흡수.
신규 — 추가
  • 없음. LLM은 완전히 동일한 입력·출력만 본다.

노트북의 역할

기존 — 복사·재사용
  • cell 3 (기존 cell 6): _qwen35_inference_params·call_llm_stream·consume_stream·BASH_TOOL·FINISH_TOOL·GOOGLE_WEB_SEARCH_TOOL. 그대로 복사.
  • cell 6 (기존 cell 10): SYSTEM_PROMPT·sub_agent_run agentic loop. 그대로 복사.
왜 복사? 기존 노트북을 손대지 않기 위해 (회귀 0). 두 노트북이 별도로 살아있다.
신규 — Sub-1 추가
  • cell 2 — 세션 시작: create_session() helper로 POST /sessions 호출. 사용자에게 URL 출력.
  • cell 4 — Bash 실행기 HTTP 라우터: parse_hwp_doc_patch_call() + run_bash_command(). hwp-doc-patch 명령을 가로채 POST /workbench로 HTTP 호출.
  • cell 5 — self-test: 라우터 단위 검증.
왜 추가? 기존 노트북은 bridge(8765) 경로였음. SoT를 서버로 옮기려면 *서버 직통 HTTP 호출*로 갈아엎어야. 노트북은 WebSocket을 *사용하지 않음* — 요청-응답 패턴이라 HTTP가 자연스러움.

웹서버의 역할 — SoT(진실 원천)

기존 — 재사용
  • server/src/main.rs — axum 서버 부트, 9개 기존 라우트(/sessions, /ops, /ir, /export …).
  • server/src/store.rs — sqlite 영속. 세션·op·snapshot 모두 여기로 기록.
  • server/src/storage.rs — minio 프록시(Sub-1에서는 비활성).
  • src/document_core/commands/edit_op.rs — EditOperation 4 variants + apply_edit_ops_json. *Sub-1의 insert_text 진짜 적용이 이 함수를 그대로 호출*.
기존 axum 라우터·DocumentCore·sqlite 영속을 모두 활용. Sub-1은 그 위에 *WebSocket 핸들러*와 *새 진입점 두 개*만 얹는다.
신규 — Sub-1 추가
  • server/src/events.rs — ServerEvent + ClientMessage enum + EventsHub(fileId별 broadcast 채널).
  • server/src/ws.rs — ws_upgrade 핸들러. WS handshake 후 `handle_socket`에서 양방향 frame loop. 클라 메시지를 받아 apply_edit_ops_json + broadcast 발행.
  • main.rs::workbench — POST /workbench. action 분기: insert_text는 진짜 적용 + Ops broadcast, 그 외는 Workbench broadcast.
  • main.rs::apply_ops 수정 — 기존 op 적용 후 broadcast 한 줄 추가.
  • Cargo.toml — axum ws feature + tokio-stream + futures.
왜 추가? 기존 서버는 *요청-응답 단방향* HTTP만 가지고 있었음. *서버 push*도, *영속 양방향 채널*도 없었음. WebSocket이 둘을 한 채널로 통합.

프론트의 역할

기존 — 재사용
  • core/wasm-bridge.ts — WASM 인스턴스 핸들. insertText·splitParagraph 등 노출.
  • hwpctl/action-registry.ts — getActionDef(id)로 액션 핸들러 조회.
  • hwpctl/actions/*.ts — 12+2 액션 클라이언트 구현. workbench 패스스루의 종착지.
  • MirrorSink 인터페이스 (session-client.ts의 외부 API) — InputHandler가 호출하는 시그니처. *Sub-1에서 유지*.
main.ts·InputHandler·hwpctl 흐름이 이미 잘 짜여 있음. 우리는 *SessionClient 내부만 갈아엎고* 외부 API는 손대지 않음 — 호출자 무변경.
신규/수정 — Sub-1
  • 수정 — core/session-client.ts: 내부를 HTTP fetch → WebSocket으로 갈아엎음. queueOp·requestSnapshot·attach·createSession은 외부 API 그대로. 내부에서 WS open·재연결 백오프·sendBuffer·onmessage 처리. onServerEvent 콜백 옵션 추가.
  • 수정 — main.ts: 기존 new SessionClient({...}) 호출 옵션에 onServerEvent 추가 — ev.kind로 분기해 ops는 WASM 직접, workbench는 hwpctl.
  • 신규 — e2e/ws-bridge.test.mjs: Puppeteer로 양방향 종단 검증.
왜 갈아엎음? 기존엔 클라→서버 미러링이 HTTP POST /ops·PUT /snapshot. SoT가 서버로 옮겨가면서 *서버→클라 push*도 필요. *같은 WS 채널로 통합*하는 게 깨끗 — 별도 SSE 채널이 따로 살지 않음. 외부 API 유지로 호출자 회귀 0.
