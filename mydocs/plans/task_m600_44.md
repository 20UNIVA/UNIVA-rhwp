# Task #m600-44 — PARA_HEADER 24 bytes 정합 (한컴 손상 알람 root cause 해소)

브랜치 `local/task44`. 사용자 보고 — rhwp-studio 편집 후 hwp 다운로드 → 한컴 편집기 "손상된 문서" 알람.

cycle 42·43 commit 후에도 한컴이 알람 띄움. CFB stream + BodyText record 정독으로 진짜 원인 식별 + 1 line fix 로 해소.

## 진단 흐름

1. **CFB stream 비교** (`examples/cfb_diff.rs`) — 수정본 FileHeader compressed=false + Preview·_LinkDoc 결함 식별 → cycle 42·43 fix.
2. **repair_A·B·C·D 검증** — 사용자 보고: 네 버전 모두 한컴에서 안 열림. 근본 원인이 더 깊다는 단서.
3. **BodyText record 비교** (`examples/dump_bodytext_records.rs`) — 수정본의 PARA_HEADER·LIST_HEADER·TABLE 모두 한컴 정답지보다 짧음.
4. **PARA_HEADER byte-level dump** (`examples/dump_para_header_bytes.rs`) — 원본 24 bytes 끝에 `00 00` (change_track_id UINT16) 추가, 수정본 22 bytes 에는 누락.

## 결함 자리

[src/serializer/body_text.rs:369-377](../../src/serializer/body_text.rs#L369-L377) 의 fallback 자리:

```rust
} else {
    // 종전 자료: instanceId(4)만 박음 → size 22 bytes
    w.write_u32(0).unwrap();
}
```

HWPX 출처 paragraph 는 `raw_header_extra` 가 비어 있어 이 fallback 분기로 들어옴. instanceId 4 bytes 만 박아 PARA_HEADER 가 22 bytes 로 끝남. 한컴 5.0.3.2+ 정답지(사업관리 참조표.hwp 포함)는 모든 PARA_HEADER 가 24 bytes (변경추적 UINT16 포함).

## Fix

```rust
} else {
    // Task #m600-44 — 한컴 정답지(사업관리 참조표.hwp 등)는 모든 PARA_HEADER가 24 bytes.
    w.write_u32(0).unwrap(); // instanceId
    w.write_u16(0).unwrap(); // change_track_id
}
```

## Cascade 효과

PARA_HEADER 가 22 → 24 bytes 로 박히자 byte offset 이 정렬되어 *그 뒤의 LIST_HEADER·TABLE record 가 잘못된 위치에서 읽히던 결함이 자동 해소*. 수정본의 LIST_HEADER 34 bytes·TABLE 26 bytes 는 *실제로는 47·28 bytes 데이터가 박혀 있지만 PARA_HEADER 2 bytes 가 짧아 reader 가 다음 record header 를 잘못된 byte 위치에서 시작했던 것*.

verify_hwpx_to_hwp_records 검증 (HWP → HWPX → HWP 경로):

| record | 종전 (cycle 44 이전) | cycle 44 후 |
|---|---|---|
| PARA_HEADER (root) | 22 | **24** ✅ |
| PARA_HEADER (cell) | 22 | **24** ✅ |
| LIST_HEADER | 34 | **47** ✅ |
| TABLE | 26 | **28** ✅ |
| CTRL_HEADER (col_def) | 38 | 38 (-9 잔존) |
