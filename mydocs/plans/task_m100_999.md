# Task #999 수행 계획서 — SSR 배포 전환: fileId 세션 기반 서버사이드 문서 영속 + 모델 조회 API

- 이슈: #999 (잠정 — gh 인증 복구 후 정식 채번)
- 브랜치: `feature/ssr` (포크: `20UNIVA/UNIVA-rhwp`, 컨트리뷰터 워크플로우)
- 마일스톤: M100 (v1.0.0)

## 1. 배경 및 목표

현재 rhwp는 **순수 프론트엔드 SPA**다. 모든 HWP 파싱/편집/렌더링이 브라우저 안 WASM(Rust)에서 일어나고, 문서 상태는 브라우저 IndexedDB+메모리에 있어 **탭/iframe을 닫으면 휘발**한다.

목표는 다음을 만족하는 **서버사이드 문서 상태 영속** 구조로 전환하는 것이다.

1. 클라이언트가 VM에 떠 있는 rhwp 에디터를 **iframe**으로 불러와 편집한다.
2. iframe 연결을 끊거나 프론트를 닫아도 **VM 서버단에 파일과 편집(patch) 정보가 유지**된다.
3. **AI 모델**이 서버 API로 해당 문서의 현재 상태를 조회할 수 있다.
4. 문서별 **`fileId`(=minio fileId) 단위 세션**으로 관리한다.

## 2. 현황 분석 (코드 근거)

| 항목 | 현재 상태 | 근거 |
|------|----------|------|
| 배포 | 정적 SPA (Vite), 백엔드 없음 | `rhwp-studio/vite.config.ts` |
| 엔진 | 브라우저 WASM(Rust). `cdylib`+`rlib` | `Cargo.toml` |
| iframe | 이미 구현 (`@rhwp/editor`, postMessage) | `npm/editor/index.js`, `rhwp-studio/src/main.ts:849` |
| 편집 IR 수정 | `&mut Document` 직접 mutate (결정적) | `src/document_core/commands/text_editing.rs:43` |
| 편집 연산 API | `insertText`/`splitParagraph` 등 명확한 연산 집합 | `src/wasm_api.rs:515-1280` |
| undo/redo | Command 양방향(execute/undo) + 스냅샷 하이브리드 | `rhwp-studio/src/engine/command.ts`, `history.ts` |
| IR→JSON 직렬화 | **없음** (HWP/HWPX 바이너리만) | `src/serializer/` |
| minio 연동 | **없음** | (전체 grep 무결과) |

**핵심 발견**: undo가 작동한다는 것은 각 편집이 이미 **역연산 정보를 보유**한다는 뜻이다(예: `DeleteTextCommand`가 삭제 전 텍스트를 `deletedText`에 보존 — `command.ts:147-160`). 따라서 편집을 **양방향 patch**로 직렬화하는 것이 구조적으로 가능하다. 단 붙여넣기/객체/표편집 등 `SnapshotCommand`(`command.ts:867-913`)는 역연산을 연산으로 표현하지 못하고 before/after 전체 스냅샷에 의존한다.

## 3. 설계 결정 (작업지시자 확정)

1. **서버**: Native Rust 서버(axum 등) 신규 구축. rhwp를 `rlib`로 의존하여 `Document` IR의 single source of truth로 둔다.
2. **동기화**: **하이브리드** — 단순/서식 편집은 `EditOperation` 양방향 patch 로그, 스냅샷형 복잡 연산은 전체 스냅샷 동기화.
3. **영속**: 작업 중에는 VM 로컬(세션 상태 + patch 로그), 확정 시 export → minio 업로드.
4. **모델 조회**: `GET /sessions/{fileId}/ir` → **Document IR JSON (신규 직렬화 경로 구현 필요)**.
5. **minio 스코프 제외**: 본 서버는 input으로 **`fileId` + hwp/hwpx 파일 바이트**만 받는다. minio 다운로드/업로드는 **외부 모듈**이 담당한다.

## 4. 목표 아키텍처

```
┌──────────────────┐           ┌──────────────────────────────────────────────┐
│  클라이언트         │           │  VM — rhwp-server (Native Rust / axum, 신규)     │
│                  │  iframe   │                                                │
│  ┌────────────┐  │◄─────────►│  세션 매니저: fileId → Document IR (메모리)         │
│  │ rhwp 에디터  │  │           │   ├─ 편집 적용: native rhwp (WASM과 동일 경로)      │
│  │ (WASM)     │──┼─ op 전송 ─►│   ├─ patch 로그(EditOperation, 양방향) 누적        │
│  └────────────┘  │ 스냅샷 PUT │   └─ VM 로컬 영속 (세션 + patch 로그)              │
└──────────────────┘           │                                                │
                               │  API:                                          │
  [외부 모듈] fileId+파일 ──────►│   POST /sessions            세션 생성/복원        │
  [AI 모델]   GET …/ir ────────►│   POST /sessions/{id}/ops    연산형 patch 적용     │
  [외부 모듈] GET …/export ────►│   PUT  /sessions/{id}/snapshot 스냅샷형 동기화      │
                               │   GET  /sessions/{id}/ir      IR JSON (모델 조회)  │
                               │   GET  /sessions/{id}/export  hwp/hwpx 바이너리     │
                               └──────────────────────────────────────────────┘
```

## 5. 스코프

**포함:**
- Document IR → JSON 직렬화 경로 (serde)
- `EditOperation` 양방향 patch 프로토콜 정의 + native 적용기
- Native Rust 서버(axum) + fileId 세션 매니저 + VM 로컬 영속
- rhwp-studio 클라이언트의 서버 미러링(연산 전송 / 스냅샷 PUT) + iframe fileId 연결
- export API (확정 저장용 바이너리 반환)

**제외:**
- minio 직접 연동(다운로드/업로드) — 외부 모듈이 처리, 본 서버는 인터페이스 경계만 정의
- 멀티 클라이언트 동시편집(CRDT/OT) — 1차는 단일 편집자 가정
- 인증/인가 — 별도 게이트웨이 가정

## 6. 단계 구성 (개요 — 세부는 구현계획서에서 3~6단계로 확정)

1. **Stage 1** — Document IR → JSON 직렬화 경로(serde) 구축 + 모델 조회의 기반 마련
2. **Stage 2** — `EditOperation` 양방향 프로토콜 정의(현 WASM API 매핑, inverse 필드 포함) + native 적용기
3. **Stage 3** — Native Rust 서버(axum) + 세션 매니저 + VM 로컬 영속(patch 로그/스냅샷)
4. **Stage 4** — rhwp-studio 클라이언트 미러링(op 전송 / 스냅샷 PUT) + iframe `fileId` 세션 연결
5. **Stage 5** — export API + 외부(minio) 모듈 인터페이스 경계 정의
6. **Stage 6** — 통합/E2E 검증 (편집→닫기→재조회→export 라운드트립, WASM↔native 결정성)

## 7. 위험 분석

| 위험 | 가능성 | 방어 |
|------|--------|------|
| WASM 편집과 native 편집 결과 불일치 | 중 | `document_core` 동일 경로 재사용, 라운드트립 회귀 테스트 |
| 스냅샷 동기화 비용(대용량 문서) | 중 | 스냅샷형 연산만 한정, 가능하면 연산형으로 흡수 |
| IR JSON 스키마 비대/버전 호환 | 중 | 모델 조회 전용 뷰 스키마 분리, 버전 필드 |
| 세션 생명주기/메모리 누수 | 중 | TTL + 디스크 스왑, 명시적 close |
| iframe↔서버 통신 채널/CORS | 중 | §8 결정 필요 |

## 8. 추가 결정 (작업지시자 확정)

1. **통신 채널**: iframe(rhwp-studio)이 VM 서버에 **직접 HTTP 호출**. (가능하면 서버가 studio 정적 자산도 서빙하여 same-origin → CORS 불필요)
2. **patch 전송 시점**: **디바운스 배치 전송** (연산을 모아 일정 간격으로 `[op, op, ...]` 전송).
3. **세션 영속 매체**: **임베디드 DB (sqlite)** — sessions/snapshots/ops 테이블.
4. **동시편집**: **단일 편집자(1차)** — fileId 세션 단위 lock. CRDT/OT는 범위 외.
