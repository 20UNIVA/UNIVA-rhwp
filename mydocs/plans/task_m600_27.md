## 진단 (cycle 27 cycle 1)

### IR 상태 — 정상

`DocumentCore::from_bytes(원본 hwp)` 직후 IR:

| 위치 | line_height |
|---|---|
| 표 cell[0].paragraphs[0].line_segs[0] | 100 |
| 표 cell[1].paragraphs[0].line_segs[0] | 2503 |
| 표 cell[2].paragraphs[0].line_segs[0] | 100 |
| paragraph 0.0.line_segs[0] (표 컨테이너 문단) | 3603 |

IR 자체는 원본 hwp 의 lineseg 값을 그대로 보존한다. paginate() 후에도 mutate 없음 — cycle 26 의 "paginate 부수효과" 가설은 *결과 부분 일치, 원인 자리 미확정*.

### export hwpx 결과 — swap 확인

`serialize_hwpx(doc)` 출력 hwpx 의 section0.xml:

| 위치 | export 의 vertsize | 원본 hwp 의 동등 위치 |
|---|---|---|
| 표 hp:tc 안 cell[0] hp:lineseg | 3603 (= paragraph 0.0 의 lh 값) | 100 |
| 표 hp:tc 안 cell[2] hp:lineseg | 3603 | 100 |
| paragraph 0.0 의 hp:linesegarray vertsize | 1000 (고정값, IR 의 3603 과 다름) | 3603 |

표 안 cell paragraph 두 곳이 *paragraph 0.0 의 lh* 로 치환되고, paragraph 0.0 자체는 *고정 1000* 으로 치환된다. 두 자리 모두 IR 의 line_segs 값을 안 쓴다.

### 직접 println 출력 — 자리 맞음

`src/serializer/hwpx/table.rs:write_sub_list` 의 reflow + lineseg 쓰기 분기 안에 println 박음 (현재 task27 분기 commit 안 된 잔재):

```
[write_sub_list] cell r=0 c=0 pi=0 reflowed_segs.len=1 first_lh=100
    -> writing hp:lineseg vertsize=100
```

write_sub_list 안에서는 IR 의 100 을 그대로 박는다고 출력한다. 그런데 export hwpx 의 동등 위치는 3603 — *write_sub_list 호출 이후 별도 path 가 같은 hp:lineseg 자리를 덮어쓴다*는 가설이 자연스럽다.

`w.get_ref().len()` 으로 byte buffer 직접 검사 시도 — generic `W: Write` 제약으로 차단. 진단 도구 한계.

### 결함 자리 — replace_first_linesegs 의 첫 매칭 오인

[src/serializer/hwpx/section.rs:65-73](../../src/serializer/hwpx/section.rs#L65-L73) 의 호출 순서:

```rust
let (first_t, first_linesegs, first_advance) = match first_para {
    Some(p) => render_paragraph_parts(p, vert_cursor, ctx),
    None => render_paragraph_parts_for_text("", vert_cursor),
};
vert_cursor = first_advance;

let mut out = EMPTY_SECTION_XML.replacen(TEXT_SLOT, &first_t, 1);
out = replace_first_linesegs(&out, &first_linesegs);
```

흐름:
1. `render_paragraph_parts(first_para)` 의 first_t 는 `render_run_content` 결과 — paragraph 0.0 안 표 컨트롤이 `render_control_slot` → `table::write_table` → `write_sub_list` 사슬로 직렬화되며 *cell 의 `<hp:linesegarray>...</hp:linesegarray>` 자료* 가 first_t 안에 박힘.
2. `out = EMPTY_SECTION_XML.replacen(TEXT_SLOT, &first_t, 1)` — 템플릿의 `<hp:t/>` 자리 (paragraph 0.0 의 linesegarray 보다 *앞쪽*) 에 first_t 박힘. 결과 — cell 의 linesegarray 들이 paragraph 0.0 의 템플릿 linesegarray 보다 *앞쪽* 에 위치.
3. `replace_first_linesegs(&out, &first_linesegs)` — `xml.find("<hp:linesegarray>")` 의 첫 매칭이 *first_t 안의 첫 cell linesegarray*. 그 자리의 내용이 first_linesegs (paragraph 0.0 의 IR lh=3603) 로 교체됨.
4. paragraph 0.0 의 원래 템플릿 linesegarray 자리는 그대로 — 정적 vertsize=1000 잔존.

결과:
- cell[0] 의 hp:lineseg vertsize 가 paragraph 0.0 의 lh (3603) 로 덮어쓰임 → row 0 두꺼움
- paragraph 0.0 자체의 linesegarray 가 *템플릿 정적 1000* 그대로 (`render_lineseg_array_fallback` 의 `push_lineseg_static` 출력값)

cell[1]·cell[2] 가 export 에서 3603 으로 보인다는 cycle 1 진단 자체는 *sub-agent 의 보조 분석 시점에서 cell[0] 자리만 결함이 명확*. cell[1]·cell[2] 의 export vertsize 는 *IR 값 (2503·100) 그대로 보존* 으로 정정.

### Fix

`replace_first_linesegs` 호출을 *TEXT_SLOT 교체보다 앞* 으로 옮긴다. first_t 가 박히기 전 상태에서는 템플릿 안 linesegarray 가 유일하므로 `find` 가 정확히 paragraph 0.0 의 자리를 매칭한다.

```rust
let mut out = replace_first_linesegs(EMPTY_SECTION_XML, &first_linesegs);
out = out.replacen(TEXT_SLOT, &first_t, 1);
```
