# Task #m600-31 — HWPX round-trip deep diff 회귀 테스트 자료

## 배경

cycle 24~30 의 fix 5건은 모두 *공통 패턴* — HWPX serializer 의 `<hp:tbl>` → `<hp:subList>` → `<hp:p>` → `<hp:run>` 4단계 안쪽 자료가 "최소 박음" 으로 멈춰 부분 스타일·자료 보존을 빠뜨림. 사용자가 시각 보고 → 디버깅하는 순서로만 catch.

기존 `ir-diff` 명령은 paragraph 단위 비교만 박음. PageDef·Table.common·cell 안쪽 paragraph 의 char_shapes·controls·line_segs 자료는 비교 범위 밖. cycle 28 (PageDef margin) 과 cycle 30 (char_shapes count) 가 자동 검출 안 된 이유.

## 자료

### fixture 3개 박음

`samples/hwpx_roundtrip/` 폴더:

| 파일 | 포맷 | 자료 |
|---|---|---|
| `baseline_business_table.hwp` | HWP | 5 paragraphs, 5 tables, 1 nested, 24 cells (cycle 25~30 의 baseline) |
| `multi_section_nested.hwpx` | HWPX | 3 sections, 84 paragraphs, 18 tables, 4 nested, 3 pictures, 673 cells |
| `pictures_equations.hwp` | HWP | 242 paragraphs, 22 tables, 14 pictures, 2 equations, 322 cells |

### deep diff snapshot 테스트

`tests/hwpx_roundtrip_deep_diff.rs`:

1. 각 fixture parse → IR_orig
2. `serialize_hwpx(IR_orig)` → re-parse → IR_hwpx (HWPX round-trip)
3. `serialize_document(IR_orig)` → re-parse → IR_hwp (HWP round-trip)
4. IR_orig vs IR_hwpx, IR_orig vs IR_hwp 의 deep diff
5. baseline 파일 (`*.baseline.txt`) 과 비교 — 처음엔 박고, 이후엔 동일하면 통과

### 비교 항목

- `PageDef.{width, height, margin_{left,right,top,bottom,header,footer,gutter}}`
- `Section.paragraphs.len()`
- `Paragraph.controls.len()` (top-level)
- `Table.common.{width, height, text_wrap, treat_as_char, vert_rel_to, horz_rel_to}`
- `Table.outer_margin_{left,right,top,bottom}`
- `Table.page_break`
- `Table.{row_count, col_count, cells.len()}`
- `Cell.paragraphs.len()`
- `Cell.paragraphs[i].{para_shape_id, char_shapes.len(), line_segs.len()}`
- `Cell.paragraphs[i].controls` — 재귀 (nested table 자료도 같은 검사)

### baseline 갱신

```bash
# 손실 자료가 줄어들어 baseline 자료를 갱신할 때
RHWP_UPDATE_BASELINES=1 cargo test --test hwpx_roundtrip_deep_diff
```

## 현재 baseline 의 손실 패턴

전부 HWPX path 에서만 발생. HWP path 는 0건.

| 자료 | 의미 |
|---|---|
| `char_shapes.len()` 손실 (예: 4 != 3) | cycle 30 fix 후에도 일부 cell 의 char_shapes 자료 1개 줄어듦 — 부분 fix |
| `controls.len()` 손실 (예: 2 != 0) | cell paragraph 의 Field·Footnote 자료 손실 (cycle 28 는 Table·Picture 만) |
| `line_segs.len()` 손실 (예: 1 != 2) | cell paragraph 의 reflow lineseg 수 차이 |
| top-level `s0.p1.controls.len() 1 != 0` | section paragraph 의 control 일부 손실 |

이 baseline 들이 *현재 알려진 손실*. 새 PR 자체 결함이 추가되면 baseline 과 diff 가 커져 `cargo test` 가 fail. fix 가 들어와 손실이 줄어들면 `RHWP_UPDATE_BASELINES=1` 으로 baseline 좁힘.
